# Test / debug / bench infrastructure audit — 2026-04-30

Audit of CSS-DOS + Calcite testing/debug/benchmarking infrastructure as
experienced by an agent walking in cold. Goal: surface friction before
any redesign decision is made. Findings are stated as observations
(verifiable) and concerns (judgement). Where I'm guessing, it's marked
explicitly.

This is a diagnostic, not a plan.

---

## Status as of 2026-05-01 — read this before acting on anything below

This audit's **factual findings still hold** — the doc drift, the four
doom benches, the three scripting DSLs, the cabinet-location chaos, the
debris, the layering survey in §11. Treat those sections as accurate
inventory.

But the **recommendations** in §10 and §11 reflect a framing that the
2026-05-01 design discussion superseded. Specifically:

- The audit talked about a "Rust adapter" / "adapter layer" as a

- Calcite's cheats (`column_drawer_fast_forward`, `rep_fast_forward`'s
  opcode-awareness, `bda_push_key`, `render_*`, etc.) are

The summary + revised plan lives in
[`audit-summary-and-plan.md`](audit-summary-and-plan.md). Read it
*after* reading this audit but *before* doing anything based on §10 or
§11's recommendations — those are partially obsolete and footnoted as

## TL;DR — the four things that matter most

1. **There is no canonical bench tool, and there's no canonical cabinet
   to bench against.** Four overlapping doom benches, three "compare"
   commands in pipeline.mjs (only one documented), three condition/
   scripting DSLs invented independently. The `doom8088.css` at repo
   root that agents grab is a stale, untracked artifact; the web bench
   builds fresh from `carts/doom8088/`; the CLI bench reads root.
   They don't compare the same cabinet by default.

2. **A scripting substrate already exists, but nothing uses it.**
   `tests/harness/lib/script-runner.mjs` + `pipeline.mjs script` is a
   JSON-based "drive a cabinet through waits/keys/screenshots" runner.
   Calcite-cli has its own native `--cond`/`--poll-stride`/
   `--script-event` DSL. The bench-doom-stages-cli.mjs uses calcite's
   DSL; the web bench-doom-stages.mjs hand-rolls its own polling in
   Playwright; the script-runner is used by no bench at all. Three
   parallel "wait until condition" implementations, none of them
   reusable.

3. **The MCP debugger is broken in this session and broadly distrusted.**
   User confirms it "started to fall apart" and agents tend to fare
   better with calcite-cli. The docs still treat MCP as the canonical
   debug surface (Agent-oriented tooling section in
   `calcite/docs/debugger.md`, harness `lib/debugger-client.mjs` wrappers,
   `tests/harness/pipeline.mjs consistency` going through the MCP path).
   Live mismatch.

4. **Documentation drift is everywhere.** Tools listed as broken aren't,
   tools claimed as working aren't, agents are pointed at deprecated
   scripts, README tables don't match source, the "Default BIOS is
   Corduroy" doesn't have a corresponding `ref-corduroy.mjs` (only
   ref-hack and ref-muslin exist), the perf logbook lives in the wrong
   repo. None of these are individually fatal but the combined effect is
   exactly the agent confusion you described.

## 1. Bench / test tools — what an agent has to choose between

### The doom benches (4 of them)

| Script | Default cart/cabinet | Mode | What it actually measures |
|---|---|---|---|
| `bench-doom-stages.mjs` | cart `doom8088-load` (alias for `carts/doom8088/`, builds in browser) | Web (Playwright + bench.html) | Full boot → in-game with stage sentinels |
| `bench-doom-stages-cli.mjs` | cabinet `doom8088.css` at **repo root** | Native (calcite-cli with `--cond`/`--poll-stride`) | Same shape, native target |
| `bench-doom-load.mjs` | cart `doom8088-load` | Web | Per user: "specifically for the loading screen" |
| `bench-doom-gameplay.mjs` | cart `doom8088-load`, post-`stage_ingame` | Web | Holds LEFT for `--window-ms`, reports simulatedFps / vramFps / paintFps |

**Observations:**

- The web/CLI pair (`bench-doom-stages` + `-cli`) is the only one called
  out as "the two benches" in `docs/agent-briefs/doom-perf-mission.md`
  and the LOGBOOK. The other two are barely mentioned (`gameplay` once,
  `load` zero times in canonical docs).
- They share none of their condition/polling/sentinel code. Each
  rolled its own.
- The web bench builds from cart **fresh** every run. The CLI bench
  reads `doom8088.css` from repo root **as-is** — last built 2026-04-28,
  2 days stale right now, no automatic rebuild. The pair is advertised
  as "honest analog" but they don't compare the same cabinet unless
  someone manually rebuilt root with the current builder. (#8)
- `doom8088.css` and its `.bios.bin` / `.disk.bin` / `.kernel.bin` /
  `.meta.json` sidecars are **not gitignored** (the gitignore covers
  `out/`, `bench/build/`, `tests/*.css`, `tests/harness/cache/`, but
  not root-level cabinets). Currently sitting untracked in the repo.

**Per user (#1, #4):** the existence of N separate bench scripts is
itself the bug. The right shape is one bench tool that takes a cart +
a profile (e.g. "boot to in-game", "loading-screen window", "in-game
gameplay window") expressed in a scripting language (Lua-style was
mentioned), with explicit polling stride. Bench-from-tick-X-to-Y or
bench-while-condition-Y-holds.

### The other doom variants

`carts/` has 7 Doom8088 variants: `doom8088`, `doom8088-cga4`,
`doom8088-cgabw`, `doom8088-m13h` (high detail), `doom8088-m13m` (medium
detail), `doom8088-t80x50` (text mode). Only `doom8088` (Mode 13h low
detail) is benched. Per user: **the others "look way worse, just around
for intrigue"** — keep but don't bench. Agents grabbing one of the
"interesting" variants by name will get no harness support. Worth a
one-liner in the cart README naming the canonical one.

### `bench/run.mjs` — old bench harness

Repo-root `bench/run.mjs` defines 3 scenarios (rogue-menu-idle,
int-heavy, mov-heavy) and writes to `bench/results.md`. Last entry
2026-04-20T08:38:29Z. Has its own `bench/build/`, `bench/carts/`. Not
referenced by CLAUDE.md, INDEX.md, TESTING.md, harness README, or the
LOGBOOK. **Per user (#5): old measure, can be removed.**

### `bench-web.mjs` — the only generic web bench

`tests/harness/bench-web.mjs` drives `/player/bench.html` for any cart
(default zork1). Documented in TESTING.md and calcite's CLAUDE.md.
This one is fine — it's the most-referenced and most-stable web bench,
positioned as the "is this fast enough to ship?" oracle.

### Profiling/diagnostic scripts (all undocumented)

Live in `tests/harness/`:

- `flamegraph-doom.mjs` — V8 sampler + tracing during doom load/ingame
- `profile-doom-load.mjs` — JS-ref-emulator-driven CALL/RET flamegraph

## 3. The CLI / bench / scripting tangle

### Three "scripted run" layers, no shared substrate

| Layer | Where | Format | Used by |
|---|---|---|---|
| **A. calcite-cli native flags** | Rust, in `crates/calcite-cli/src/main.rs` | `--cond NAME:ADDR=VAL[,...]:then=spam:KEY:every=N`, `--script-event TICK:KIND[:ARGS]`, `--script-file PATH`, `--poll-stride N` | `bench-doom-stages-cli.mjs` |
| **B. JS script-runner** | `tests/harness/lib/script-runner.mjs` + `pipeline.mjs script` | JSON array of `{wait, type, key, shoot}` steps | `pipeline.mjs script` only — no bench uses it |
| **C. Per-bench inlined polling** | Each `bench-doom-*.mjs` rolls its own | Hand-coded JS polling in Playwright callback | All web benches |

Layer A is documented only inside calcite-cli's clap help. The harness
README and TESTING.md don't mention `--cond` / `--poll-stride` /
`--script-event` (TESTING.md mentions `--script-event` once in passing).

This is the clearest example of "agents add another layer instead of
cleaning up." Per user (#1, #4, #12): **a single scripting layer
(Lua-style was suggested) that subsumes all three is the right shape**
— bench from tick X to Y, run while condition holds, set explicit polling
stride, don't poll every tick.

### `calcite-bench` vs `calcite-cli`

Two binaries that overlap:

| | `calcite-cli` | `calcite-bench` |
|---|---|---|
| Run a cabinet for N ticks | yes | yes |
| Halt on memory address | `--halt` | `--halt` |
| Restore from snapshot | `--restore` | `--restore` |
| Profile output | implicit (`--op-profile`) | `--profile`, `--phase-breakdown`, `--histogram`, `--iterations` |
| Keyboard / video / scripts | yes | no |
| Diagnostic dumps (`--dump-mem-range`, `--sample-cs-ip`) | yes | no |
| Used by | bench-doom-stages-cli, fast-shoot, --dump-* invocations | LOGBOOK references for op-distribution + phase profiling |

Per user (#1): no real reason for two to exist. Benching without
keyboard input isn't useful for real-cart workloads. The calcite-bench
binary is documented in `../calcite/docs/benchmarking.md` as "primary
tool", but in practice `bench-doom-stages-cli.mjs` (the perf benchmark
of record per LOGBOOK) goes through `calcite-cli`. Calcite-bench's
unique features (`--profile`, `--phase-breakdown`, `--iterations`,
`--histogram`) could fold into calcite-cli. **Open question:** keep
calcite-bench at all, or merge into calcite-cli + the eventual
scripting layer?

### Probes — 20 of them

`crates/calcite-cli/src/bin/probe_*.rs` and `probe-*.rs`:

- 20 binaries in `target/release/`, 14 declared in Cargo.toml `[[bin]]`,
  the other 6 auto-discovered from `src/bin/probe_*.rs`.
- Naming inconsistent: `probe_byte_periods` (underscore, declared) vs
  `probe-bytecode-shape` (hyphen, declared) vs `probe_bif_pairs`
  (underscore, auto-discovered).
- Per user (#18): these are transient diagnostic one-shots from
  individual perf investigations (splash-fill Apr 14-18, fusion Apr
  28-29, byte-period detector Apr 28). Should be deleted; git history
  preserves them if anyone needs them again.
- `probe_splash_*` (5 probes): from the Apr-14 splash investigation
  documented in calcite/docs/log.md — investigation done, probes never
  referenced again.

## 4. Debugger landscape — broken in practice, central in docs

Per user (#12, #13): **MCP debugger is currently broken; agents fare
better with calcite-cli**. This is a load-bearing fact that the docs
don't reflect.

### What the docs say

- `calcite/docs/debugger.md` is the canonical reference; treats MCP as
  the primary surface. Has an "Agent-oriented tooling" section listing
  ~15 MCP tools (`inspect_packed_cell`, `compare_paths`, `watchpoint`,
  async `run_until`, `trace_property`, etc.).
- `docs/debugging/calcite-debugger.md` quick-reference points there.
- `tests/harness/README.md` and `TESTING.md` lean heavily on
  `pipeline.mjs consistency` / `triage` / `fulldiff` — all of which go
  through `lib/debugger-client.mjs` → MCP debugger.

### What's actually configured / running

- `.mcp.json` in CSS-DOS root points at
  `C:/Users/.../calcite/tools/mcp-shim.mjs`. Daemon expected on
  `127.0.0.1:3334`.
- `mcp-shim.mjs` claims autostart was deliberately removed ("the
  operator is in charge: start the daemon explicitly").
- `start-debugger-daemon.bat` says the opposite ("mcp-shim.mjs
  auto-starts the daemon on first client connect, so you rarely need to
  run this manually").
- In this session, no `mcp__calcite-debugger__*` tools are available to
  me — confirms user's #13: currently broken (daemon not running, or
  shim+daemon contract mismatch).
- `target/release/calcite-debugger.exe.old` from Apr 23 — debris from
  `kill-and-rebuild.bat`. Per user (#14): probably stale, can be removed.

### Two flavors of debugger session

1. **Ad-hoc HTTP** — `cargo run -p calcite-debugger -- -i cabinet.css`,
   port 3333. Documented as the "Quick start" in calcite/docs/debugger.md.
2. **Persistent MCP daemon** — `calcite-debugger.exe --listen
   127.0.0.1:3334`, started via `start-debugger-daemon.bat`, accessed
   via mcp-shim.mjs. Documented elsewhere.

Same binary, two operating modes. No doc says "these are two ways to
run the same thing; pick the first for ad-hoc debugging, pick the
second if you want MCP tools in Claude Code." Agents discovering one or
the other wouldn't realize the other exists.

### `pipeline.mjs consistency`/`triage`/`fulldiff` go through the broken path

These all spawn a calcite-debugger child via `lib/debugger-client.mjs`.
Per user "MCP started to fall apart" — same code path. **TESTING.md
points agents at these tools as the main "find first divergence" oracle.**
If the underlying debugger is unreliable, the harness wrappers are too,
even if the harness wrapper layer itself is fine.

### Per user: scripting subsumes both

Per #12: a proper scripting layer would replace the debugger's
`watchpoint`/`run_until_*`/`trace_property` use cases (which is what
agents actually want from MCP) with calcite-cli `--cond`-style
predicates expressed in a real scripting language. `inspect_packed_cell`
and `compare_paths` are different and remain debugger-shaped, but they
matter less when the bench harness already exposes "land at tick N,
dump state, compare."

That's fine.

The conspicuously-missing `ref-corduroy.mjs` doesn't matter if
`ref-machine.mjs` handles all flavors via sidecars (which it does — I
verified `ref-machine.mjs` doesn't care about BIOS flavor).

| `calcite-core/`, `calcite-wasm/` (web target) | `../calcite/web/pkg/calcite_wasm_bg.wasm` | `wasm-pack build crates/calcite-wasm --target web --out-dir ../../web/pkg`, OR dev server `_reset` endpoint |
| `calcite-core/`, `calcite-cli/` (native target) | `target/release/calcite-cli.exe` etc | `cargo build --release` from calcite root |
| `calcite-core/` (debugger target, native) | `target/release/calcite-debugger.exe` | `kill-and-rebuild.bat` (because daemon usually holds it open) |

**Cache layers an agent needs to be aware of:**

- Browser HTTP cache (dev.mjs sets no-cache, fine)
- Browser Cache Storage (player uses it for cabinet bytes)
- Service Worker (player registers one)
- WASM module cache (browser)
- Calcite's load-time compile cache (in-memory only, per-tab)

The dev server's `/_reset` endpoint wipes WASM + prebake; the `/_clear`
HTML page clears browser-side state. Both are documented in dev.mjs's
own comments and **nowhere else**. Agents working purely from CLAUDE.md
+ docs won't know to use them.

**Per user (#15, #16): wants a clear explicit process documented for
each.** Suggested doc location: a new `docs/build-flow.md` or
`docs/rebuild-when.md` consolidating the table above + the `/_reset`
endpoint + the `kill-and-rebuild.bat` flow.

## 11. Layering audit — survey of where calcite cheats

**Scope note (revised 2026-05-01):** this section is a **survey** of
calcite's cardinal-rule violations, **deliberately preserved as
knowledge for a future cleanup project**. The cleanup itself is **not
in scope** for this audit's follow-on work. Calcite's cheats stay
where they are for now; the cleanup work focuses on the genuinely
mislocated CSS-DOS-shaped code in calcite (Severity 1 below — moves
and deletions only, no API design).

The cardinal rule: **calcite knows nothing above the CSS layer.** The
operational test for any piece of code: *could a calcite engineer who
has never seen a CPU emulator, read Kiln, or know what a cabinet
contains derive this rule by staring at CSS shape alone?* Yes → calcite/.
No → CSS-DOS/.

There are two categories of code:

- **Calcite-proper** — generic CSS-evaluation machinery. Parser,
  evaluator, pattern recognisers that fire on structural CSS shape, the
  state model, snapshot/restore.
- **CSS-DOS-the-platform** — kiln, BIOSes, builder, carts, player,
  player shim (SW), bench harness.

The cabinet `.css` is the contract between them.

**There is no third "adapter" category.** Earlier framings of this
audit posited an adapter layer (Rust code that hooks into calcite to
inject upstream knowledge). That was a category error: an "adapter"
that registers REP patterns into calcite is just calcite's cheating
relocated, not eliminated. The cleanup direction for calcite's cheats
is **calcite recognises shapes generically from CSS structure**, with
no upstream awareness anywhere — same direction as the existing
`byte_period` and `fusion_sim` substrate. If a shape can't be
recognised generically, the pressure goes onto kiln to emit a more
recognisable shape.

This survey flags concrete cases. Items are roughly in severity order:
violations of the no-cross-repo-dep rule first (cleanup IS in scope —
these are file moves), then "calcite knows about upstream" leakage
(survey only — out of scope), then naming/borderline cases.

### Severity 1 — calcite has hard CSS-DOS dependencies (CLEANUP IN SCOPE)

These violate calcite's CLAUDE.md ("There is no crate-level dependency
between the two repos and there must never be one — calcite's only
interface to CSS-DOS is the `.css` output").

**These are in scope for the cleanup work** — see summary doc Chunks B
and C. They're file deletions and moves, no API design required.

**`calcite/crates/calcite-cli/src/menu.rs`** invokes
`node ../CSS-DOS/builder/build.mjs` to build `.com`/`.exe` programs into
cabinets when the user picks them from the interactive menu. ~30 lines
in `resolve_to_css()` shell out to CSS-DOS's builder. Hardcoded relative
path `../CSS-DOS/builder/build.mjs` and falls back to a parent-dir
search for a `CSS-DOS` sibling.

This is a direct cross-repo dep. The interactive menu was built to
"discover programs and offer to play them" — but the *act of building
a cart into a cabinet* is a CSS-DOS responsibility, not a calcite one.
The fix is: calcite-cli's interactive menu only lists pre-built `.css`
files. If the user wants to build something, they invoke the CSS-DOS
builder themselves (or a CSS-DOS-side tool wraps "discover + build +
launch calcite-cli").

**`calcite/serve.mjs`** is described in its own header as "CSS-DOS dev
server — serves the site and generates CSS from .com/.exe uploads."
Lives in calcite, references `CSS_DOS_DIR`, exists to drive a CSS-DOS
workflow. Should be in CSS-DOS, or deleted (CSS-DOS already has
`web/scripts/dev.mjs` doing the same job better).

**`calcite/site/`** is a complete (if old) CSS-DOS web UI:
`<title>CSS-DOS</title>`, "CSS-DOS v0.1" branding, `program.html`,
`run.html`, custom logo. Distinct from `calcite/web/` (the calcite-only
demo). This is a former CSS-DOS frontend that ended up living in
calcite. CSS-DOS now has `player/` and `web/`. Calcite's site/ is
either dead code or a parallel-history fork that no longer matters.

**`calcite/programs/`** has 20+ `.com`/`.exe` files (rogue, frotz,
bootle, eliza, etc.). These are guest x86 programs. Per cardinal rule
they don't belong in calcite at all. They're consumed by `menu.rs` /
`serve.mjs` / `run-*.bat`. If those go, programs/ goes.

**`calcite/output/`** has 20+ pre-built `.css` cabinets. Per user
(#10): super out of date. Calcite's CLAUDE.md treats them as
"benchmark fixtures". They ARE adapter artifacts (cabinets built from
specific CSS-DOS carts at a specific time). Belong in CSS-DOS, or
nowhere.

**`calcite/run-bios-test.bat`, `run-oldbios-test.bat`, `bench-splash.bat`,
`run-splash.bat`, `run-web.bat`, `run-js.bat`** — most reference BIOS,
mode 13h, bootle.com/bootle-ctest, and CSS-DOS-side worktrees. Should be
in CSS-DOS or deleted.

### Severity 2 — calcite-core has upstream knowledge baked in (SURVEY ONLY — OUT OF SCOPE)

These don't cross repo boundaries, but they violate the "could a
calcite engineer derive this from CSS shape alone" test. They're inside
the engine.

**Out of scope for the cleanup work.** Documented here so a future
calcite-cheats-cleanup project knows where to look. The cleanup
direction is "calcite recognises these shapes generically" or "kiln
emits more-recognisable shapes" — NOT "relocate the cheat to a Rust
adapter."

**`calcite/crates/calcite-core/src/compile.rs::column_drawer_fast_forward`**
(lines ~5780-5990). A 21-byte literal `COLUMN_DRAWER_BODY` const
matching the Doom8088 column-drawer x86 asm. The function name, the
21-byte sequence, the per-iteration semantic comment block ("mov al,
dh; shr al, 1; mov bx, si; xlat; ..."), and the post-iteration state
update (advance DI by 0xF0, etc.) all encode "this is Doom's column
drawer". The comment claims "Cabinet-agnostic — any cabinet whose ROM
happens to contain this sequence" but no other cabinet will.

The LOGBOOK explicitly documents this as a **dead lead**
(2026-04-29: "fusion FFD: dead end on this window … fusion belongs in
another cabinet's perf budget"). It's still in main as of 2026-05-01,
gated behind `CALCITE_FUSION_FASTFWD=1` env var. **Per the cardinal
rule and the logbook's own conclusion, this should be reverted.** The
generic substrate it sits on (byte_period, fusion_sim) is fine; the
specific Doom-shaped consumer is the violation.

**`calcite/crates/calcite-core/src/compile.rs::rep_fast_forward`** —
the REP STOSB/MOVSW/etc. fast-forward path. Hardcodes x86 opcodes by
hex value (0xAA STOSB, 0xAB STOSW, 0xA4 MOVSB, 0xA5 MOVSW, 0xA6 CMPSB,
…). Reads slots named `--opcode`, `--hasREP`, `--repType`,
`--segOverrideValue` — all upstream-named. Panics with messages like
"STOS/MOVS with REPNE prefix — extend rep_fast_forward to handle
repType=2 for write-side string ops".

This one is harder. REP fast-forward is real and necessary for
performance. But the implementation is fundamentally x86-aware; it
isn't pattern-recognising any "general repeated body" — it's
recognising specific x86 REP variants by opcode byte.

The clean direction (future work, not in scope here): calcite
recognises the *CSS shape* that REP loops compile to — generically,
from structural CSS analysis, with no opcode awareness. If kiln's
emitted CSS doesn't lend itself to such generic recognition, that's
pressure on **kiln to emit a recognisable shape**, not pressure to
relocate the opcode-aware code into an adapter.

(Earlier framings of this audit suggested decomposing rep_fast_forward
into a `BulkPattern` registry exposed by calcite, with x86-specific
patterns registered from CSS-DOS-side. That was the wrong direction
— it would just move the cheat one level out without eliminating it.
Discarded.)

**`calcite/crates/calcite-core/src/eval.rs:1404`** — hardcodes
"keyboard / `__1keyboard` / `__2keyboard` property → linear address
0x500" with the comment "The CSS-DOS BIOS polls address 0x500 (word)
for keystrokes." Pure adapter knowledge inside the eval loop.

**`calcite/crates/calcite-core/src/state.rs::bda_push_key`** — hardcodes
the IBM PC BDA keyboard ring buffer layout (0x41A head, 0x41C tail,
0x480 buf start, 0x482 buf end). Method on calcite's State that knows
the BIOS Data Area is at 0x400 and has these specific offsets.

**`calcite/crates/calcite-core/src/state.rs::render_screen` /
`render_screen_ansi` / `render_screen_html` / `render_framebuffer`** —
take `(base_addr, width, height)` as args, but the *interpretation* of
the bytes is x86-IBM-PC-specific: CP437 character set, 2-bytes-per-cell
char/attr layout, attribute bit encoding (`BBBFFFFF`), CGA palette for
mode-13h-style framebuffers. These are renderers for x86 video memory
shapes wearing a "generic memory rendering" coat.

**Renderers belong in CSS-DOS.** The CSS-DOS-side `tools/render-mode13.mjs`
and `tools/gen-vga-dac.mjs` already exist; they're the right home. The
calcite-core versions are a parallel implementation.

**`calcite/crates/calcite-core/src/state.rs::bulk_copy_bytes`** —
generic in shape but the docstring is entirely about REP MOVSB
semantics, DF=1, UPX self-overlapping decompression. Implementation
genuinely is generic byte-copy; doc string is upstream. Easy fix:
rename or rephrase. Borderline.

**`calcite/crates/calcite-core/src/state.rs:361` — `read_mem16` fast
path** — gated on `packed_cell_size == 2`. Comment: "pack == 2 (the
only value emitted today)". Generic-looking but only fires on
CSS-DOS's specific pack choice. Might just need `>= 2` or a more
careful generalisation.

**Slot-name knowledge throughout `calcite-core`**: 77 references to
"x86" / "8086" / "opcode" / "BIOS" / "CSS-DOS" / "column_drawer" /
"mode13" / "framebuffer" in `compile.rs` alone. Most are comments
("dominant pattern in CSS-DOS bytecode", "covers the BIOS pattern
`style(--opcode: N) and style(--uOp: K)`") — these *describe the
shape calcite is recognising* but justify it via upstream context.
Comment-style violations rather than code-style; easier to fix.

### Severity 3 — CSS-DOS-shaped code that ended up in calcite (CLEANUP IN SCOPE)

These are CSS-DOS-side files that physically live in `calcite/`. Pure
file moves; no API change. In scope for Chunk C.

**`calcite/web/video-modes.mjs`** is the clearest case. Header
literally says "Video-mode decode table for CSS-DOS players." References
BDA 0x0449, 0x04F2, 0x04F3, kiln. Imported by both `calcite/web/
calcite-worker.js` (calcite's own demo) and `CSS-DOS/player/
calcite-bridge.js` (the actual CSS-DOS player) — the latter via the
dev-server alias `/calcite/video-modes.mjs`.

This is pure CSS-DOS adapter code that ended up in calcite because two
consumers wanted it and one of them happened to be in calcite. Move to
CSS-DOS (`web/lib/video-modes.mjs` or similar); calcite's demo can
import it via its own copy or via its own simpler demo-only renderer.

**`calcite/tests/fixtures/x86css-*.css`/`html`, `dos-games/`,
`chrome-baseline.json`** — x86CSS test fixtures with `--AX`, `--BX`,
etc. properties; `.com` files and their compiled `.css`. Calcite's
CLAUDE.md acknowledges these as "Pre-compiled CSS imported from
CSS-DOS." Adapter-layer artifacts. CSS-DOS-side would be more honest;
they only exist because calcite's primitive tests want a "real CSS"
input.

### Severity 4 — branding leakage (SURVEY ONLY)

**`calcite/crates/calcite-cli/src/cssdos_logo.rs`** + assets —
CSS-DOS branding inside calcite-cli, called from the menu's program
picker. Decorative; minor; but reflects "calcite-cli is Doom8088's
runner" mental model that the cardinal rule explicitly rejects.

### What CSS-DOS has that might belong in calcite

I checked. **Not much.** The pieces I considered:

- `tools/js8086.js` — generic 8086 emulator. Already used by the
  conformance harness as a reference. Could in theory live in calcite
  (calcite has its own conformance.rs), but it's not a CSS evaluator,
  it's an x86 reference — belongs as a CSS-DOS conformance asset.
- `kiln/` — entirely upstream-knowing. Right repo.
- `tools/peripherals.mjs`, `tools/lib/bios-handlers.mjs`,
  `tools/lib/bios-symbols.mjs` — i8259 PIC, NASM symbol parsing, BIOS
  INT handlers. Upstream. Right repo.
- `web/` (player, bridge, prebake, dev server) — CSS-DOS adapter +
  delivery. Right repo.
- `tests/harness/lib/cabinet-header.mjs` — parses the
  `/*!HARNESS v1!*/` JSON block. CSS-DOS adapter convention. Right
  repo. (Verified: calcite-core does not read this header.)

The asymmetry is real: calcite has lots of CSS-DOS in it; CSS-DOS has
~nothing of calcite's that's misplaced. That's roughly because calcite
predates CSS-DOS-the-platform and accreted upstream knowledge as it
became the engine for one specific cabinet. The cleanup is mostly
one-way: pull CSS-DOS-shaped code out of calcite.

### What this means for the work in scope

The cleanup work touches **only the Severity 1 and Severity 3 items**
above — pure file moves and deletions, no calcite-core API changes.
See summary doc Chunks B and C.

In short:

- **Delete from calcite/:** `site/`, `serve.mjs`, `serve.py`,
  `programs/`, `output/`, the CSS-DOS-shaped `.bat` files, the
  BROKEN-marked `tools/*` files.
- **Move from calcite/ to CSS-DOS/:** `web/video-modes.mjs` (the
  one piece of unambiguously-CSS-DOS-shaped code in calcite/web/).
- **Strip calcite-cli/src/menu.rs:** remove the
  `node ../CSS-DOS/builder/build.mjs` shell-out. Menu lists pre-built
  `.css` only.

Severity 2 (calcite-core's internal cheats) and Severity 4 (branding)
stay where they are. They're catalogued for a future calcite-cleanup
project; doing them now would conflate scope.

### What's left for the future calcite-cleanup project

For when someone takes on calcite's cardinal-rule cleanup as its own
project:

- `column_drawer_fast_forward` — the logbook authorises a revert; do
  it then. Generic substrate (`byte_period`, `fusion_sim`) stays.
- `rep_fast_forward` — the hardest one. Direction: generic CSS-shape
  recognition, OR kiln emits a more recognisable shape. Not "register
  patterns from outside."
- `render_screen*` / `render_framebuffer` — move out of calcite-core,
  but the destination is CSS-DOS-side rendering code (the player's
  shim or the bench page), not "an adapter."
- `bda_push_key`, the `--keyboard` → 0x500 mapping, the BDA layout
  knowledge — same: out of calcite-core, into CSS-DOS-side code that
  uses calcite's public memory-write API.
- `--cond` DSL's `vram_text:NEEDLE` — replaced by Chunk D's generic
  predicates; consumers express VRAM-text-matching as
  `byte_pattern_at(0xB8000, stride=2, needle)` with the upstream
  knowledge living on the consumer side.
